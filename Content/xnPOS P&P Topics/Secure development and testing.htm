Secure development and testing policy

Purpose

The purpose of this Policy is to ensure that the risk of security vulnerabilities being introduced into XPS production environments is minimised.

As part of the XPS Systems (“XPS”) commitment to the security of its clients’ data, XPS has implemented this Policy to ensure that all software that is developed by XPS is developed (coded) in line with industry best practice and tested for security vulnerabilities. This is in order to ensure that vulnerabilities are not inadvertently or maliciously introduced into XPS live production environment – which could potentially lead to XPS data being compromised. This Policy contains the minimum requirement for the development and testing of software deployed on XPS systems.

Scope

This Policy applies to anyone who works for XPS, including permanent employees, fixed term employees, agency workers, temporary staff, consultants or contractors and parties (also known as Data Users), that have any involvement (development, coding or testing) with code that is deployed within XPS hosted application environments.

The scope of thisstandard is limited to the deployment of code that is to be deployed on systems that are located within XPS hosted application environments.

Controls

General

All of XPS software development is developed in-house. Thus, XPS requires a full Software Development Life Cycle (SDLC), and must ensure that its internal development team codes in line with industry best practice and incorporate information security throughout the SDLC, where that software is to be deployed within XPS hosted application environments.

All code and software that is to be deployed into XPS hosted application environments must be developed in-house.

All application developers that add and make changes to code within XPS must be trained in secure coding techniques. Including the following areas as a minimum:

  1. Injection Flaws
  2. Buffer Overflows
  3. Insecure cryptographic storage
  4. Insecure communications
  5. Improper error handling
  6. All high risk vulnerabilities identified by the XPS risk management process

This development procedure must be formally communicated to all staff that develop code and software that is to be deployed into XPS hosted application environments.

The communication of this procedure is the responsibility of the Lead Application Development Manager, who must ensure that this is undertaken prior to employment, on an annual basis and after any change to this procedure.

Employees must formally accept and acknowledge that they will develop software, maintain coding standards and perform testing in accordance with this procedure.

XPS must follow the process described in this document that details how all code deployed into the hosted application environment will be tested for security vulnerabilities before accepting it from an application developer.

This procedure only applies to code that will be deployed within XPS hosted application environments and from this point forward all statements only apply to that scope.

Software development and testing

XPS recognises that the internal applications team develops code on its behalf and has a SDLC which includes information security throughout the life cycle. XPS also recognises and tries to be prescriptive as to which industry best practices are used, and realises that additional complexities to the development process maybe incurred. However, XPS requires reassurance that the code that it deploys on its systems is developed and tested in line with industry best practice, including information security throughout the life cycle, to reduce the risk of code being deployed on those systems that contains security vulnerabilities. Thus, each application developer must formally accept and acknowledge that its SDLC meets the requirements defined in this section and provide evidence to XPS, to describe how these requirements are met. The evidence supplied will be used by XPS for auditing purposes:

All software maintenance and development must be based on industry best practices and fully documented in a SDLC. The SDLC must:

  1. Define how information security is included and considered throughout the software development process.
  2. State which industry best practices are followed.
  3. Define how applications are developed in accordance with PCI DSS.
  4. State that all software changes or the reconfiguration of software must be fully tested prior to release into the XPS live environment.
  5. State that any test or development environment that is to be released into the XPS live environment, must have all data and test accounts removed prior to release and that it is not permitted to have data or accounts in the XPS release. This includes application accounts, usernames or passwords.
  6. State that there must be a peer review of code developed on behalf of XPS before it is released to XPS. Manual and / or automated code review processes are used as defined within the Change Control Policy.
  7. State that the code review process:
    • Covers all application code changes.
    • Requires that the code is reviewed by persons different to the original author.
    • Requires that the code is reviewed by persons who are knowledgeable in code review techniques.
    • Requires that the code is reviewed by persons who are knowledgeable in secure coding practices.
    • Ensures that the secure coding guidelines defined in the SDLC are followed.
    • Ensures any code corrections identified must be rectified prior to release to the XPS live environment.
    • Ensures that code review results are reviewed and approved by management prior to release.
  8. State that there must be separate development and testing environments with access control in place to enforce the separation, and separation of duties must be enforced to ensure no single individual has access to both environments.
  9. State that the development and test environments must be segmented using either routers with stringent access control lists or firewalls with stringent access control lists.
  10. State that separation of duties between test and development environments must be strictly enforced.
  11. State that user access lists to test and development must be maintained.
  12. State that it is prohibited to use genuine payment card data in any development or test environment and that test card data shall be used for such purposes.
  13. Define the test data removal procedure that ensures that any test or development environment that is released to XPS must have all test data and test accounts removed prior to deployment.
  14. State that it is not permitted to have test data or test accounts in the environment to be release to XPS - this includes test application accounts, usernames or passwords.
  15. Define the specific code conventions and styles used for software development.
  16. State that all code will be commented as it is written and that software developers are responsible for creating the program documentation.
  17. Define how all web applications developed by XPS, must follow industry accepted secure coding guidelines. All web applications within the scope of PCI DSS whether internal or external are covered by this requirement.
  18. Define how the selected secure coding guidelines for the development of all web applications within the scope of PCI DSS must be followed.
  19. Describe how all application developers must be trained on the selected secure coding guidelines.
  20. State that a documented web application development process must be used and that all web-facing developed code must be reviewed in order to remove any vulnerability as outlined in the OWASP Top Ten (www.owasp.org). This document must define the processes for identifying the following vulnerabilities:
    • Injection flaws (including SQL injection)
    • Buffer overflows
    • Insecure cryptographic storage
    • Insecure communications
    • Information leakage and improper error handling
    • Cross-site scripting (XSS)
    • Improper access control
    • Cross-site request forgery
    • Input validation
    • Session fixation
    • Insecure direct object references
    • Remote file inclusion
    • PHP security
  21. Define how the web application development process will ensure that web applications are not vulnerable to ‘high’ risk vulnerabilities.
  22. State that all web-facing application code must be reviewed for common vulnerabilities via manual or automated application vulnerability security assessment tool prior to release to the XPS live environment.

Code management procedures

XPS uses in-house software developers to develop its code, and is clear to these employees that it requires that code is developed using an SDLC. Thus, strong management of those code reviews is required in order that code is delivered that meets the agreed standard. This must include appropriate tests to ensure that none of the security vulnerabilities that the developers’ SDLC seeks to eliminate are present in the code that is to be placed into the live environment.

Procedure: Code procurement
  1. Initial requests for Code development or changes must be directed via the Design Team.
  2. Once the development or change has passed through the Design Team and has been approved, it will pass through the technical design team who must identify any PCI DSS implications and formally record whether the development or change will impact on the hosted application environments.
  3. If the development or change does not impact the HOSTED APPLICATION ENVIRONMENTS, then complying with XPS Secure Software Development Procedure is optional, but recommended as best practise.
  4. The Technical Design Team (in conjunction with any project team) must then contact the relevant application developer with the requirements for development or change in order that the code is procured.
  5. Only employees that are part of the application development team may be used to make changes to application code.
Procedure: Code development

The Application Development Team and IT Infrastructure Team (if applicable) must then work with the application developer, within the developer’s SDLC until such a time that the code is ready for delivery to XPS live environment.

 Procedure: Maintain a secure code gateway
  1. The Application Development Team must take delivery of the code from the developer via a secure method (i.e. not sent as an unencrypted email attachment).
  2. Once the code is received by the Application Development Team, it must hand this to the Lead Application Development Manager for security testing.
  3. The Lead Application Development Manager must arrange for testing of the code and be satisfied that it does not contain any security vulnerabilities.
  4. The testing of the code may be either by an automated and / or manual method. After any upgrade, significant modification or infrastructure change an internal penetration test must be conducted.
  5. The person or persons performing the security testing of the code must be knowledgeable in code review techniques.
  6. The person or persons performing the security testing of the code must knowledgeable in secure coding practices.
  7. The testing of the code must confirm that:
    • All data has been removed from the release (including test card data).
    • All test data has been removed from the release.
    • All test accounts have been removed from the release (including application accounts, usernames or passwords).
    • The specific code conventions and styles that have been agreed with the application developer have been adhered to.
    • Appropriate commenting is contained within the source code.
    • Web-facing code does not contain any vulnerability that is outlined in the OWASP Top Ten (www.owasp.org).
    • Web-facing code does not contain any injection flaws (including SQL injection).
    • Web-facing code does not allow buffer overflows.
    • Web-facing code does not allow insecure cryptographic storage.
    • Web-facing code does not allow insecure communications.
    • Web-facing code does not allow information leakage.
    • Web-facing code properly handles errors.
    • Web-facing code does not allow cross-site scripting (XSS).
    • Web-facing code properly manages access control.
    • Web-facing code does not allow cross-site request forgery.
    • Web-facing code correctly validates input.
    • Web-facing code does not allow session fixation.
    • Web-facing code does not contain any insecure direct object references.
    • Web-facing code does not allow remote file inclusion.
    • Web-facing code does not contain any PHP security flaws.
    • Web-facing code is not vulnerable to ‘high’ risk issues as per PCI DSS requirement 6.2.
  8. The person or persons performing the security testing must produce a report to detail the testing undertaken and the results of those tests.
  9. The report must be presented to the Lead Application Development Manager for review and approval.
  10. If the report details security vulnerabilities that require remediation, then this must be passed back to the Application Development Team – in which case the Application Development Team must follow steps 5.1.4 onwards.
 Procedure: Releasing code in the live environment
  1. The Application Development Team must request that the code is put live via XPS change control process (see XPS Change Control Process).
  2. When requesting the change, the Application Development Team must submit the security testing report as accompanying evidence.
  3. Any web-facing code must be placed in the live environment under the protection of an application layer firewall. For all public web applications address new threats and vulnerabilities on an ongoing basis and ensure these applications are protected against known attacks by either of the following methods, Review public facing web applications via manual or automated vulnerability security assessment tools or methods, at least annually and after any changes.

Exceptions

As per the Information Security Policies, any policy exceptions must be formally documented and presented for review and sign-off by the Information Security Manager. All such exceptions are valid for a period of no longer than one year, upon which they will be subject to further review and formal acceptance.

Enforcement

Breach of this policy may result in the termination of contract and possible further disciplinary and/or legal action.
