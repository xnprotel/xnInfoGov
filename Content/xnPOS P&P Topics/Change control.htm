Purpose

The purpose of this policy is to control the risk of security vulnerabilities being introduced into XPS production environments.

Scope

This policy applies to anyone who works for XPS, including permanent employees, fixed term employees, agency workers, temporary staff, consultants or contractors, who may need to request, evaluate, review or make a change.

The scope of this policy is limited to changes to hardware or software in the hosted application environment, whether this be a new installation, an upgrade/change to an existing system, or the decommissioning of a system This will include, but is not limited to, the following:

  1. Firewall, switch and router changes.

  2. Implementation of new hardware in the hosted application environment or the upgrading of existing hardware.

  3. Software changes in the hosted application environment including new implementations and patches/upgrades to existing software.
  4. System configuration changes.
  5. Any software changes in the payment gateway.
  6. Decommissioning/removal of hardware/software within the hosted application environment.

Tasks that are outside the initial scope of this policy include those changes made within the daily administrative process. Examples of daily administrative tasks are:

  1. Password resets
  2. User additions, deletes and modifications
  3. Rebooting machines when there is no change to the configuration of the system
  4. File permission changes

Preamble

All network environments change over time, whether the change is planned or unplanned. Change Control Policies help to minimise the inadvertent introduction of security vulnerabilities when implementing planned, unplanned, or recovery changes to a network environment.

Change control is essential in all environments. A change in a staging or development environment that creates a design flaw will typically be replicated to the production environment. Lack of change control policies for all environments can cause flawed configuration changes or code enhancements in the production environment.

Another danger to production environments is disaster recovery situations. If changes made to a production environment are not properly documented, as per a change control policy, systems that have been recovered might be put into production without having been properly hardened. Without proper change control procedures in place, systems could be placed back into production after planned maintenance has been performed, without having been properly secured.

TThis policy specifically addresses changes made to Xn protel Systems (“XPS”) hosted application environments.

Controls

Requests for software or hardware

Any request to change or implement new software or hardware in the hosted application environment must be submitted in line with XPS ‘Change Management Procedure’. This includes the submission of a Change Control Form which contains the following information:

  1. Description of change requested
  2. Reason for request
  3. Documentation of impact
  4. Time frame for implementation
  5. Back-out procedures
  6. Functionality testing to verify that the change does not adversely impact the security of the system.
  7. Management approval for the change

Requests for software development

Where any requested change relates to software development including software modification and security patches then XPS ‘Secure Software Development and Testing Policy’ will also apply. The following policies apply to software development changes:

  1. Any development/test environment must be separate from the production environment.
  2. Software code changes must be reviewed by individuals other than the originating code owner, and by individuals who are knowledgeable in code review techniques and secure coding practices. Code reviews must always ensure that code is developed per secure coding guidelines.
  3. Following software code reviews, appropriate corrections must be implemented prior to being pushed into the live environment, and code review results must always be approved by management prior to release.
  4. Access control must be in place to enforce the separation between the development/test environment and the production environment.
  5. There must be a separation of duties between personnel assigned to the development/test environments and those assigned to the production environment.
  6. Live payment (credit/debit) card numbers must not be used in the development and testing environments.
  7. All test data and accounts must be removed before production systems become active.

Exceptions

As per the Information Security Policies, any policy exceptions must be formally documented and presented for review and sign-off by the Information Security Manager. All such exceptions are valid for a period of no longer than one year, upon which they will be subject to further review and formal acceptance.

Enforcement

Breach of this policy may result in the termination of contract and possible further disciplinary and/or legal action.
